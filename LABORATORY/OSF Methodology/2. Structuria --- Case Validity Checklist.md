# Structuria Case Validity Checklist — Working Standard

**Status:** Formalized July 19, 2026, in response to a specific, named risk: hand-picking candidate cases is higher quality than bulk mining, but introduces a harder-to-detect failure mode — selection bias that looks like judgment rather than a process error. This checklist exists to operationalize "valid" before any case is read closely enough to trigger motivated reasoning about it.

**Governing principle, already on record elsewhere in this project and directly on point here:** *law alignment is confirmation, not classification.* A case is coded on its own structural merits first. Whether it happens to support an existing invariant is checked afterward, not used to decide whether the case counts.

---

## 1. Pre-Selection Discipline (before reading anything closely)

**Search using structurally-neutral terms only, when searching at all.** Where systematic archive traversal is possible — going through an ordered set or structured sample regardless of content — prefer it over search-term-based selection entirely. It doesn't depend on avoiding biased terms, since it isn't selecting by term at all. Reserve neutral-term search for cases where full or sampled traversal isn't practical.

**For personally-sourced cases, priming cannot be prevented — disclose it instead.** A day's events are not chosen from a search; they're whatever happened, filtered only by "conflict, failure, or high energy interaction." What was recently discussed, read, or thought about before a specific memory surfaced as a candidate is a real, unavoidable influence. Log it, the same way an externally-sourced pick gets a one-line reason: *"this came to mind after [X]."* This does not remove the priming. It makes it visible enough to weigh later, which is the only thing actually achievable here.

**Set a batch size before looking, not after.** Decide how many candidates will be reviewed in a sitting — e.g., ten — before reading the first one. Evaluate all ten as a batch. Stopping as soon as a satisfying case appears, rather than at the pre-set number, is the single easiest way for this checklist to be defeated without anyone noticing, including the person doing it.

**Log the pick reason before reading past the first paragraph.** One sentence: what made this candidate get opened rather than skipped. This is not analysis — it's a bias audit trail. If a pattern emerges over time in *why* things get picked (always dramatic, always short, always about work), that pattern is worth knowing about independent of whether any individual case is valid.

---

## 2. Structural Validity Gate (applied blind to which invariant it might support)

Run in this order. A case that fails any gate stops there — later gates don't rescue it.

1. **Is this a real, dated or datable event — not a general reflection, a hypothetical, or a composite the narrator is already presenting as illustrative?** If it's not describing something that actually happened, it can't be Tier 2-equivalent evidence, regardless of how well it fits the taxonomy.
2. **Is there enough concrete, specific detail to code at all?** A vague mood or a general life philosophy isn't a case. If the only way to make it fit is to supply the missing specifics yourself, it fails here.
3. **Test the case against all five domains before settling on one.** Which of Judgment, Boundaries, Cohesion, Stewardship, or Peril could this plausibly be, on a first read, before deciding? Record more than one if more than one applies — this is the step that most directly enforces "classification before confirmation."
4. **For whichever domain seems to fit, does the case satisfy every required structural element, not just one evocative detail?** For Peril specifically: insulation that was actually functioning, contact that actually happened, loss of control, and a revelation — all four, not a story that only has the dramatic ending.

**4a. For self-authored cases specifically: does each element clear on the written record alone, or only with the help of memory that isn't written down anywhere?** Externally-sourced material can only ever be judged on what's actually on the page — no narrator to consult, no memory to fall back on, which is exactly why external cases feel more effortful to evaluate than self-authored ones. That effort gap is not evidence the checklist is working better on self-authored material — it's evidence self-authored material has been quietly exempted from a constraint external material can't avoid. A required element that's only true because you happen to remember it, rather than because the write-up actually states it, is clearing on privileged access, not on evidence. If an element wouldn't survive a stranger reading only the raw observation, it doesn't survive here either — go back and make it explicit in the write-up, or don't count it as cleared.
5. **Adversarial reframe: can this case be fully explained without the physics claim at all?** If a mundane account — bad luck, ordinary difficulty, a decision someone made — explains it completely, it fails, even if it also loosely resembles the domain.

**5a. For cases that arrive with their own built-in expert commentary (AHRQ PSNet, NASA APPEL, and similar sources), read and classify the raw event first, in writing, before opening the commentary at all.** This is not a formality — the commentary section is exactly the mundane, established-field explanation adversarial reframe already asks for, and reading it first would let it quietly shape the classification instead of testing it. Sequence: read the event, run the full structural gate blind, write down the resulting classification, then open the commentary and check it against what's already written. Convergence is real, if partial, confirming evidence — not proof Structuria found something new, since the expert field may have reached the same conclusion independently, on its own terms. A more conventional explanation in the commentary that the written classification doesn't survive is adversarial reframe working correctly, not a failure to force past. A written classification exists specifically so this comparison checks against something fixed, not against a memory that's easy to revise once the expert's own account is already in view.
6. **Would this classification survive being checked by someone who doesn't know what you were hoping to find?** This is the gate the two-pass process exists to actually test, not just assert.

## 3. Exposure & Consent Gate

Applies after structural validity, since there's no reason to run an exposure check on a case that isn't usable anyway.

- **Nested, non-participant check, per Publication & Anonymization Standard §7.** Is the load-bearing content of this case actually about someone other than the narrator, who never consented to anything, at any distance? If yes, excluded — no further review needed.
- **Consent-scope check.** Does the platform's own stated purpose plausibly cover this kind of downstream analytical use? Still an open question for StoryCorps specifically, not resolved by this checklist — see the Publication Standard for the standing recommendation to ask the platform directly rather than assume.
- **Assign an Exposure Tier (A/B/C)** per the existing standard, same as every case in the internal corpus.

## 4. Evidentiary Tier — open design question, not yet resolved

A hand-picked, externally-sourced, independently-written real story doesn't cleanly fit any tier in the existing Evidentiary Tier System. It's not the author's own field observation (Tier 2). It's not institutional news coverage (Tier 3). It's arguably closer to Tier 1's original spirit — *"independent of the author's own account or interpretation"* — than anything else, since the narrator wrote it for reasons that have nothing to do with this project, before anyone here ever saw it. But Tier 1 was defined around a researcher actively observing events in real time, not retroactively reading an already-written story. This is worth a real decision rather than a default: either fold this into Tier 1 with a noted subtype, or name a new tier. Flagging rather than resolving, since this changes how much weight every case selected this way will carry.

**Tested, not resolved, July 13, 2026 — see the Blind Provenance Test methodology document.** A blind test comparing genuine and AI-fabricated accounts was run specifically to inform this question, and it doesn't settle it. The test measured authorship class (human versus AI-generated); this open question is actually about epistemic reliability (how much an unverified account can be trusted regardless of who wrote it) — a real human narrator is exactly as capable of the self-narrative-substitution error this project has already found twice in its own corpus (CASE-041, CASE-046) as a fabricated one is. One real, if cautionary, finding did transfer: the structural validity gate showed no discriminating power in that test, which argues for more conservatism in this tier assignment, not less — clearing the Checklist should not by itself be read as evidence of reliability for externally-sourced material. This question remains genuinely open, and is more likely to be resolved by processing an actual StoryCorps case than by further abstract testing.

**A third, distinct variant identified July 15, 2026: expert-mediated case studies (AHRQ PSNet, NASA APPEL), not to be conflated with the raw-narrative question above.** Both are external and not the author's own observation, matching the surface shape of the StoryCorps question — but the underlying evidentiary character is different in kind, not degree. StoryCorps stories are raw and unanalyzed; nothing has been structurally interpreted before Structuria touches them. AHRQ and NASA cases arrive pre-analyzed by an established professional tradition — patient safety science, program-management lessons-learned — before any Structuria-specific reading occurs. This cuts two ways rather than simply strengthening the case: real expert scrutiny has already been applied, which raw narrative never receives, but any Structuria reading that merely restates an already-published expert conclusion in different vocabulary is not independent evidence for this project's own claims — it's borrowed legitimacy from a field that already reached the same conclusion on its own terms. NASA APPEL's historically documented, retrospectively-analyzed cases may fit better under the proposed-but-unused Tier 4 (historical example) than under any Tier 1 framing; AHRQ's shorter, more clinically-immediate case-and-commentary format may sit closer to Tier 1's spirit despite the same expert-mediation caveat. Neither has been formally assigned. Flagged as its own decision, not folded into the StoryCorps question above, and not resolved by anything already tested.

## 5. Two-Pass Process

1. **Pass one (author):** run §§1–4 on a candidate. Record the result, including anything that failed and why.
2. **Pass two (Claude), blind.** Raw case text only, no framing, no indication of which domain or invariant it was selected for. Independent classification against §2, run cold.
3. **Compare.** Agreement is mild positive evidence the classification is real rather than motivated. Disagreement is not a tiebreak to resolve in favor of whichever answer feels right — it's the same diagnostic signal the original three-way Claude/Gemini/Copilot disagreement was: evidence the case is more ambiguous than it looked, worth sitting with rather than forcing to a verdict.

---

## 6. What This Checklist Does Not Solve

It reduces the risk that a case gets accepted because it confirms something. It does not eliminate the deeper selection bias in *which platform, which search terms, which day* someone goes looking in the first place. That's a real, remaining limitation, worth remembering rather than treating this checklist as a complete answer to the bias problem it was built for.
