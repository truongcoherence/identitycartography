# Structuria Pattern Comparison Protocol — Working Standard (Draft)

**Status:** Proposed July 12, 2026. Updated July 19, 2026. Governs what happens after a case has cleared the Case Validity Checklist — how a validated case actually updates the physics it's checked against. This is the bottom-up method itself, extracted from precedent set across this session's restoration work.

---

## 0. Analysis vs. Interpretation — the Honest Limit of This Protocol

**Analysis:** anything checkable against an explicit, stated criterion. Two analysts, same case, same criteria, should reach the same answer — disagreement here means the criterion wasn't explicit enough, not that there's a legitimate difference of view.

**Interpretation:** everything else — which of several plausible domains a case belongs to, whether a pattern is a variant or genuinely new, how much a piece of corroboration should weigh. This is exactly where the original underdetermination event happened: three AI systems, identical rules, three coherent and mutually exclusive readings of the same field note. Self-consistency checks (Compression, Independent Derivation, run by one analyst on their own conclusion) do not catch this — each of the three original models could have passed those checks on its own reading, because each was internally coherent with itself. That's the actual gap in this protocol's earlier draft.

**No procedure eliminates the interpretation zone.** Classifying real human behavior into a finite set of categories always carries some irreducible judgment. This protocol does not claim to prevent underdetermination. It commits to four specific mechanisms for making it visible and checkable instead of letting it hide inside a confident-sounding single answer — §§1.1, 1.2, 2.5, and 2.6 below.

---

## 1. The Four Possible Outcomes

**A. Confirms an existing invariant, primitive, constraint, distortion or other concept.** Independently, not reused evidence. Cross-context corroboration counts for more than same-context repetition.

**B. Reveals a variant of an existing primitive, constraint, distortion or other concept.** Fold the refinement into the existing narrative, flagged as a refinement, not a new source count.

**C. Suggests a genuinely new pattern.** Requires an explicit statement of _why_ no existing concept covers this, and the candidate is held, not canonized, until a second, independent, non-reused case corroborates it. **The bar is two, and this is a floor, not saturation.** Two cases is what's actually happened before every promotion made this session — it is not a principled claim that two is always enough. Grounded theory's own stopping rule (keep sampling until new cases stop revealing new properties of the category) is more honest about this; adopt the honesty even without adopting the full apparatus. If a category keeps producing new dimensions past its second case, it isn't done just because it cleared the floor.

**1.1 — Outcome C additionally requires a documented, defeated alternative.** Before any new invariant is proposed, the strongest competing domain reading must be written down explicitly and explained as worse, not silently passed over. "This fits Peril" is not sufficient; "this fits Peril rather than Judgment or Cohesion, specifically because [named reason]" is. This is the direct fix for the original failure: the three divergent models never had to confront or defeat each other's readings — each just produced its own. Writing the losing alternative down, every time, forces that confrontation onto the page instead of leaving it implicit.

**1.2 — Constant comparison, backward as well as forward.** When a case sharpens or refines an existing invariant (Outcome B), check whether any _previously filed_ case under that invariant needs re-reading in light of the sharper definition — not just whether the new case fits. This will usually change nothing. It should still be checked every time a definition moves, not assumed safe by default. Borrowed directly from grounded theory's constant comparative method, which this protocol was missing.

**D. Contradicts an existing invariant.** If a case actively shows an existing invariant false rather than merely not fitting, the invariant is re-examined with full five-test rigor, and if the contradiction holds, retired outright — documented as _retired, contradicted by evidence_, distinct from _unproven_. Precedent: Stewardship's Custodial Authority.

## 2. Comparison Procedure

1. **Identify candidate domain(s), the Meta-Concept Layer, or the Architecture of Signal and Outcome** via the Case Validity Checklist's §2.3, extended to seven options rather than five. A case that doesn't cleanly fit within one of the five IDPs should also be checked against the Meta-Concept Layer and the Architecture of Signal and Outcome directly, not only against domain-level invariants — Law 01 and Orientation were both missed by domain-only checking for a real stretch of this project's history before being recognized as cross-domain, and CASE-043/CASE-044 were both processed against the Architecture document's own prohibitions well before this option was ever named explicitly here. **Caveat, not yet resolved:** the Meta-Concept Layer itself currently has unaudited entries beyond Law 01 (at minimum Law 02 and Law 03, per the Unified Structural Registry) that have not received the same scrutiny Law 01 received before being trusted as a stable comparison target. Treat Law 02/03 as provisional reference points until that audit happens, not as settled ground on par with Law 01.
    
2. **Compare against each existing invariant's definition individually.** A case can fail every existing invariant and still belong to the domain (Outcome C).
    
3. **Run Compression on the comparison:** would the invariant's current strength actually change if this case were removed?
    
4. **Run Independent Derivation:** would someone reach the same invariant-match without being told which invariant this case was gathered to test?
    
5. **When a case plausibly fits more than one domain, do not resolve by preference or resonance.** Two required steps, not one:
    
    **2.5 — Run Substitution explicitly between the competing domains, not just within one.** Take the case's central claim and test it against both candidate invariants' definitions side by side. If either definition can absorb the case with no loss of fit, the categories themselves may not be discriminating enough at this boundary — that's a finding about the taxonomy, not just about this case, and should be flagged as such rather than silently resolved by picking one.
    
    **2.6 — For any case being used to justify a new invariant (Outcome C) or retire an existing one (Outcome D), get a second, independent classification before finalizing — not a self-check, an actual second party.** This applies at the point of promotion — when a held candidate is actually being moved to formal invariant status, or when an existing invariant is actually being retired — not at the moment a candidate is first proposed and held. It is run against the retained raw observation, not a real-time parallel log, since requiring live dual-documentation on every candidate is not sustainable on high-volume days and isn't necessary if the raw material is preserved for later independent reading. Raw case only, no framing, same discipline as the Case Validity Checklist's two-pass process. If the two classifications agree, proceed. If they diverge, the divergence itself is the result — log it as an open, contested case (per the precedent in Cohesion §5 and Peril §5) rather than adjudicating a winner. A single analyst's confidence, however well-reasoned, is not sufficient evidence to settle a question that has already once been shown to produce coherent, contradictory answers from equally confident sources.

## 3. What This Protocol Does Not Cover

It does not tell you _where_ to find candidate cases (Case Validity Checklist §1) or how to format one once it's placed (Case Note Processing Standard). It only governs what a validated case is allowed to do to the canon once it's in front of you. It does not, and cannot, guarantee that two careful analysts will always agree — it only guarantees that when they don't, the disagreement is recorded rather than buried under whichever answer sounded most confident.

## 4. Held Candidates Have No Fixed Sunset — By Design, Not By Oversight

Falsification (Outcome D) is immediate: the moment a countercase disproves an existing invariant, it is re-examined and, if the contradiction holds, retired — no waiting period. But a held Outcome C candidate, still awaiting its second source, has no controllable timeline, because there is no way to schedule when a naturally-occurring second instance will appear. Setting a fixed expiration would force one of two bad outcomes: premature canonization under deadline pressure, or deliberately manufacturing a test scenario to force resolution. The second is worse than the first — a scenario constructed specifically to test a candidate is no longer naturalistic evidence; the act of engineering the test contaminates the very observation it produces, the same distortion risk that makes controlled-experiment designs a poor fit for this kind of naturalistic, field-sourced claim. Held candidates therefore persist without expiration, reviewed periodically for continued relevance whenever the Case Registry itself is updated, never resolved by manufactured circumstance.

## 5. Contribution Is Not Sufficiency — A Required Safeguard, Not Optional Caution

**Law 01 protects against an unwarranted moral verdict when something goes wrong — substituting blame for a structural account. This safeguard protects against the mirror-image failure: substituting an unwarranted structural account for legitimate credit when something goes right.** Same shape, opposite direction, and it needs its own explicit rule rather than assuming Law 01 already covers it.

**The specific inferential error this guards against:** finding that a structural factor contributed to a positive outcome is never, by itself, evidence that a personal factor did not. These are different claims. Almost every real outcome has multiple genuine causes operating together — discovering one says nothing about whether another was also present. Concluding "structure was present, therefore the achievement wasn't earned" without something that actually discriminates between "structure alone was sufficient" and "structure plus real capability together produced this" is not structural analysis. It's a moral verdict wearing structural language, the exact category error this entire methodology exists to catch, just running toward discredit instead of toward blame.

**The required test before any finding may reduce or qualify a person's earned credit:** genuine discriminating evidence — comparable to CASE-046's completion-rate gap, not merely "a structural factor can be identified." Absent that kind of evidence, the correct output is "a structural factor contributed; whether it was independently sufficient is undetermined" — never a discrediting conclusion by default. The default assumption, when evidence doesn't discriminate, is combined contribution, not sole structural causation.

This applies with equal force in the Case Validity Checklist's structural gate and in Outcomes A–D above: a positive-outcome case being processed for a domain like Judgment or Stewardship — or for the Meta-Concept Layer or the Architecture of Signal and Outcome, which run through the same classification mechanism — must clear this same bar before any structural finding is permitted to imply the person's own contribution was diminished.

## 6. Canon-Integration Fidelity — A Missing Check, Identified July 14, 2026

**Every other transition in this project's pipeline has something watching it. The step between "here's what a classification concluded" and "here's what the canon document ends up saying" did not, until this section.** Raw observation to case file has the transcription addendum, the Specificity Check, and the Reconstruction Distortion Risk document. Case file to classification has this Protocol's own Outcomes and defeated-alternative requirement. Canon to publication has the Publication & Anonymization Standard. Nothing previously checked whether the written provenance table entry actually matches what the classification found, rather than something modestly stronger.

**This is not hypothetical — it already happened once, and was caught during the July 14 canon audit.** Boundaries invariant (e) was described as "quadruple-sourced." Two of the four cited cases were the same incidents already counted as sources for invariant (d) — genuinely independent sourcing for (e) specifically was two cases, not four. The classification itself may have been entirely sound; the drift occurred specifically in how confidently the write-up described it.

**The check, stated plainly:** before any provenance-table entry or "double/triple/quadruple-sourced" claim is finalized, verify that every case counted toward it is independent evidence _for that specific claim_ — not reused from another invariant's count, and not counted twice under two different names for the same underlying incident. This is a mechanical check, not a judgment call: it can be run by tracing each cited Case ID back to every other place in the corpus it's already cited, the same way the Case Registry's own reuse-flag was designed to be checked, but applied at the moment a claim's language is written, not only when a registry entry happens to be reviewed later.

**Scope:** applies to every domain's provenance table, not only where a problem has already been found. Boundaries (e) is confirmed; the other domains have not yet been re-checked against this specific standard and should not be assumed clean by default.
