# Structuria Specificity Check

**Status:** July 14, 2026. A standalone, operational test, distinct from the Reconstruction Distortion Risk document, which examines the danger of running this check at the wrong time. This document is the tool itself: what it asks, when to apply it, and how to apply it to an actual draft rather than as a vague background worry.

**When to run this:** only after a complete, unfiltered draft already exists, per the Reconstruction Distortion Risk document's sequencing requirement. Never while a memory is still being recalled. Ideally run by someone other than whoever produced the draft.

---

## 1. The Core Question

Would this specific detail have been written down even if the author didn't already know what it needed to prove?

Not a question about the case as a whole. A question asked of each individual detail that happens to be doing real work for whatever invariant the case sources.

## 2. The Operational Version, What Actually Distinguishes a Real Tell From a False Alarm

Unusual precision, by itself, is not suspicious. This project's own blind provenance test found the opposite pattern in one of its clearest results: genuine memory often carries oddly specific, non-load-bearing detail, the kind that doesn't serve the story at all. That kind of precision is a marker of authenticity, not a warning sign.

The actual red flag is a combination of two things occurring in the same detail, not precision alone:

1. The detail is unusually specific.
2. The detail is doing exactly the interpretive work a required structural element needs, no more, no less.

A detail that is precise and incidental, serving nothing, resolving nothing, is the good kind of specificity, the kind that survives scrutiny. A detail that is precise and load-bearing, existing only because the claim needs it to exist, is the kind worth stopping on.

## 3. Worked Example, Using the Case That Prompted This Document

CASE-002's canon description reportedly carried more specificity, naming a particular department pairing and a regulatory-interpretation framing, than its underlying raw note independently supports. Run through the test above: is this the harmless kind of precision, or the load-bearing kind? It is load-bearing. The added specificity does not sit alongside the story as incidental color; it directly supplies the exact distinction the invariant needs to be satisfied. That combination, precise, and precise in exactly the place the claim needed precision, is what should have triggered a stop, whether or not it was consciously noticed at the time.

## 4. How to Apply This to a New Draft

1. Read the complete draft once, in full, without stopping to evaluate anything.

2. Identify every detail doing real work for whichever invariant or constraint the case is meant to source.

3. For each one, ask: is this detail's specificity necessary to the claim, or would the claim survive this detail being vaguer, or absent entirely?

4. If the claim would not survive the detail being vaguer, if the precision is exactly and only what the claim needed, flag it.

5. A flagged detail is not automatically false. It requires one further step: can this specific precision be independently corroborated (a raw note, an external record, a second person's account), or does it rest entirely on the same recall that produced the draft? If the latter, the detail should be marked as unverified precision in the case file, not silently treated as confirmed.

## 5. What This Does Not Do

It does not distinguish a true memory from a false one. It only identifies where a draft's precision and a claim's needs suspiciously coincide, which is a signal worth checking further, not a verdict on its own. A flagged detail may still be entirely accurate. This test only says it needs a second form of support before it can be trusted at the same level as a detail that never needed flagging in the first place.
