# Structuria Case Note Processing Standard — Working Standard

**Status:** Finalized July 12, 2026. Originally proposed as a draft, extracted from the implicit pattern present across four exemplar case files (CASE-001, CASE-004, CASE-009, CASE-028). Governs how an already-confirmed-valid case gets turned into a `CASE-NNN.md` file — distinct from the Case Validity Checklist, which governs whether a _new_ case should enter the corpus at all. This document assumes that question is already settled; every case in scope here is currently sitting, already trusted, inside one of the five restored canon documents.

**No longer draft, and specifically because it's been tested rather than only reasoned about.** The template held up under direct use on the corpus's two hardest cases — CASE-028 and CASE-031 — including refinements that only surfaced from that use: the Notes field split into Structural and Editorial, Anonymization Status marked permanently internal-only for a structural reason (the reversal-key risk), and the baseline anonymization defaults (subject-substitution, degendering) proven at zero evidentiary cost on both cases. One of those two tests passed by accident rather than by design — a drafting error on CASE-031 went unnoticed until direct correction, and the case read identically either way. That's a stronger confirmation than deliberate testing alone would have produced, and it's the basis for closing this out rather than leaving it open indefinitely.

---

## 1. Fixed Schema

Every case file uses the same six fields, in this order:

1. **Title** — `# CASE-NNN — [Descriptive Name]`, name matching the Case Registry exactly.
2. **Header block** — Evidentiary Tier, Exposure Tier (with one-line justification, not just the letter), Fieldwork Index cross-reference (or "None"), Status (Active source case / Contrast case / Borderline, not used / Held open).
3. **Narrative** — full case detail. Any correction or refinement made after the case was first written (the "several mentor figures, not one" example) is merged directly into the narrative, not appended as a separate patch note — the narrative should always read as the current best account, not a diff.
4. **Cited By** — every invariant or constraint, across every domain, that references this case, with its role (primary source / corroborating / contrast case / secondary reference) and an explicit reuse flag if the same case supports more than one domain's separate claim.
5. **Anonymization Status** — which Publication Standard §6 transformation technique, if any, has actually been applied — not just which tier it's been assigned.
6. **Anonymization Status — internal-only, always, structurally, not by convention.** Unlike Notes, this field has no public-eligible version under any circumstances. Its entire purpose is documenting what transformation was applied to identifying information — which means its content necessarily discusses the original, unaltered form alongside the anonymized one. If this field were ever visible alongside a public-facing case, it would function as a reversal key: stating outright that "the subject" refers to "the author" undoes the substitution it describes. The record of an anonymization technique must be at least as protected as the identity it removes, or the protection doesn't actually hold. This field never moves to public form, regardless of a case's own exposure tier.
7. **Notes — split into two explicitly scoped parts, not one mixed field.** This field's prior definition mixed public-eligible content with internal-only content under one label, exactly the category error the Publication Standard exists to prevent, just one level down. Resolved as follows:
    - **Structural Notes (public-eligible):** why the case is structurally illustrative — the kind of thing worth keeping if this case ever moves toward public form. Default to including only what a reader would need to understand the case's role, nothing about the corpus's own internal history.
    - **Editorial Notes (internal-only, always):** handling flags, cross-references to other internal-only documents (the running Meta-Architecture document, session-specific context), anything about the case's personal significance to the author, or instructions to future editors. Never eligible for public form regardless of the case's own exposure tier — this is process guidance about maintaining the corpus, not case content.

## 2. Workflow, Per Case

1. Locate the case's current narrative inside its source canon document(s), using the Case Registry ID and name.
2. Extract the full narrative faithfully — not summarized, not condensed. If a case has been refined or corrected in conversation since it was first written, fold the correction directly into the extracted narrative.
3. Assign Evidentiary Tier based on the case's own intrinsic properties (cued vs. uncued, author's own experience vs. third-party, high-stakes-retrospective vs. high-frequency-low-stakes) — this is a property of the case, not of whichever invariant currently cites it.
4. Assign Exposure Tier by running the Recognizability Test explicitly, not by inheriting whatever tier was informally assigned during the earlier corpus-wide pass — that pass was itself a first draft.
5. **Build the Cited By section by searching all five canon documents for the Case ID, mechanically** — not by relying on memory of which invariants use it. This is the same fix already applied once this session: a search, not a read-through.
6. Note Anonymization Status honestly — most pending cases will read "not yet transformed," and that's the correct, expected answer for anything above Tier A.
7. **Before finalizing, run the Pre-Publication Mechanical Check (Publication Standard §10) against the new file specifically** — name search, "mentor" without "figure," institution/program specificity — the same check, applied to new output, not assumed unnecessary because the content came from an already-approved source.
8. Update the Case Registry status column from "Pending" to "Drafted."

## 3. Batching

Process in small batches, not all observations at once — a batch of five or six, followed by a spot check of the mechanical check step, before continuing to the next batch. This is slower than one pass through everything, but it means a systemic template error (if one exists) surfaces after five files, not after thirty-two.

## 4. Transcription Fidelity — Addendum, July 14, 2026

**The question this addendum answers: if Claude drafts the narrative for a case file, does that compromise the data's status as real?** No, and the reasoning matters more than the answer — whether an event happened is fixed by the event itself, before any drafting occurs. What actually varies is whether the _written record_ faithfully represents what was observed, which is a transcription-fidelity question, not a reality question. This addendum gives that question the same discipline §4a of the Case Validity Checklist already gives to evaluation: checkable against the source, not trusted because it reads well.

**Two categories, kept visibly distinct in every drafted narrative:**

- **Transcription** — content drawn directly from the raw observation, staying as close to the author's own words and phrasing as the material allows. No invented connective tissue, no smoothing over gaps the raw note doesn't fill.
- **Analysis** — anything added during drafting that wasn't in the raw note: noting a case touches more than one domain, naming a structural pattern the author didn't name himself, supplying context that clarifies without appearing in the original. Marked explicitly as analysis, not folded silently into the account of what happened.

**Required before any drafted case counts as the authoritative record:** the author checks the draft against the actual raw note, not just against whether the draft sounds accurate. This is not a new mechanism — it is the same one that caught CASE-002's specificity drift during the July 14 audit, applied prospectively to drafting instead of retrospectively to auditing. It worked once already. This addendum exists to make sure it runs every time, not only when something happens to prompt a second look.

**What this does not solve, and shouldn't be mistaken for solving:** whether the underlying raw material exists at all for a given case. That is a separate, prior question — this addendum governs the fidelity of a transcription once raw material is in hand, not what to do when it isn't.

**A further distinction, for cases where no original note can be located at all: pure retrospective reconstruction is a weaker act than transcription, and must be labeled as such, not folded into the same category.** Transcription checks a draft against an external record. Reconstruction from memory alone has nothing external to check against — the same risk this project already recognizes for high-stakes retrospective accounts generally, sharpened here by one specific factor: reconstructing a case that already has a known destination (a specific invariant it's cited to support) carries real motivated-reconstruction risk, not hypothetical risk. Any case file built this way must state plainly that no original transcript was located and the account is retrospective recall, dated approximately rather than precisely, and should be treated with the same caution this project already applies to any single-source, high-stakes retrospective material — not silently upgraded to read like a contemporaneous note.
