# Structuria — Independent Replication Guide

**Status:** July 14, 2026. Written for any reader, using any AI system, who wants to pull this project's publicly available methodology and run the same analytical process independently. Deliberately platform-agnostic and deliberately independent of the current specific content of the five domains — that content will keep changing; the process described here shouldn't need rewriting every time it does.

**Before anything else: this project's own history is the reason this guide exists in its current form, not a footnote to it.** Three different AI systems, given identical source material and identical rules, once produced three different, internally coherent, mutually incompatible conclusions. That event is why this methodology insists on independent verification rather than trusting any single analysis, regardless of which AI performs it. A reader using this guide should expect the same thing to happen to them, and should build in the same check.

---

## 1. What to Pull — By Category, Not By Filename

### **1. The methodology documents. Pull these regardless of anything else.** 

These describe the process itself, not any specific finding, and are the most durable, version-independent part of this corpus: how a candidate case is screened for validity, how it's transcribed into a usable record, how it's compared against existing claims, how it's checked for exposure risk before any public use, and how a claim at any level — a single term, a single invariant, a claim about how domains interact, a claim about a felt signal — can be shown false. 


Look for the current versions of: an evidentiary grading system, a case validity checklist, a case documentation standard, a pattern comparison or classification protocol, a publication and anonymization standard, and a falsifiability framework. Names may vary by the time you read this; the function each one serves should not.

 ***These documents can be found in the formal methodology component on OSF or in the Laboratory directory on GitHub.***

### **2. The current domain canon — pulled as "whatever is current," not as a fixed target.**  
 
 The specific claims here will have changed by the time you read this, possibly substantially. Pull the most recent version available, and do not assume any individual claim is more confirmed than its own stated status says it is.

 ***These documents can be found in the Structuria Active Canon directory on GitHub.***

### **3. The status and audit documents — read alongside the canon, never instead of it.** 

This project maintains a running, honest account of which claims are fully verified, which are real but not yet independently checked, and which have no real evidentiary basis yet. A reader who only reads the domain canon and skips this layer will come away more confident than the material has actually earned.

 ***These documents can be found in the Change Log directory on GitHub or the files of the OSF project.***

#### **4. The honest-limits documents.**

A statement of this project's own evidentiary limitations, in specific terms rather than general hedging, and a record of what's been tried, retired, and why. Read these before forming a view of how much weight to put on anything else.


 ***These documents are accounted for by the first three category. The Boundary Conditions and Falsifiability Document, Development History, and Limits and Invitation to Extend explain the limits honestly. The case registry and case notes is the substrate that embed these limits***

## 2. The Cross-AI Reliability Warning, Stated Directly

Different AI systems are not interchangeable evaluators, and this project has direct, tested evidence of why. A blind test run by this project's own author, comparing genuine and AI-fabricated narrative across three different systems, found that each model has distinguishable default tendencies — some systems' fabricated narrative was easier to identify than others', for reasons specific to that system's typical output patterns. This means a classification produced by one AI system should not be assumed to generalize to what a different system would conclude from the same material, and it should not be assumed that any single system's read is authoritative.

**Practical consequence: run the same two-pass discipline this project already uses internally.** Have one AI perform an initial classification. Have a second, genuinely independent instance — a different system if possible, or at minimum a separate session with no memory of the first pass — perform the same classification blind, with no framing about what the first pass concluded. 
Agreement across independent, dissimilar evaluators is real evidence. Disagreement is not a failure to resolve by picking whichever answer sounds more confident — it's diagnostic information about how ambiguous the material actually is.

## 3. Reading Order

1. The honest-limits documents — establishes what confidence level is actually warranted before reading anything else.
2. The status and audit documents — establishes which specific claims currently carry that confidence and which don't.
3. The methodology documents — establishes the process to apply to any new material.
4. The current domain canon — read last, as an example of the process's output at this moment in time, not as a fixed target to defer to.

## 4. What Publicly Available Actually Means, as of This Writing

Not everything described above is necessarily public yet at the moment you're reading this. Methodology is the intended first layer of public release for this project, ahead of case-level detail, which requires its own anonymization process before any public use. Check whatever the current, live listing actually contains rather than assuming this guide's description of "what should be there" matches "what is there today."

## 5. What This Guide Does Not Claim

It does not claim that following this process with any AI system will produce the same conclusions this project has reached internally. It claims the opposite is entirely possible, and that this is expected, not a failure — the whole reason independent verification matters is that it might not agree, and finding out is the point.
